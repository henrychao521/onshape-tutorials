用 AI 自動化 3D CAD 建模
業界現況、方法與成功案例

調查「有沒有人用 AI 協助 Onshape／其他 CAD 建模、用什麼方法」。結論直指：我們實測卡住的「點 WebGL 畫布」是業界刻意繞開的死路。

★一句話結論

業界與社群已收斂到同一條路：AI →「參數化程式碼 / REST API」，不是 AI 去點 GUI／畫布。本專案實測卡住的「用視覺精密點 WebGL 畫布」正是大家刻意繞開的死路。視覺的正確用途是『驗證』（算圖＋比對），不是『操作』（點選）。Onshape、Blender、Zoo、學術四個獨立領域證據完全收斂到此範式。

①Onshape 專屬

社群已有多個實際可用的方案，全部走 REST API 程式化建模、API 金鑰認證、掛在 Claude Code。

②其他 CAD

同一個「程式碼 / API」範式，橫跨商用與學術。

  • Zoo.dev Text-to-CAD ✓ 生產級 API — agentic 程式（Plan→Act→Observe→Repeat，像 Claude Code），產自家語言 KCL 程式碼＋參數化 B-Rep（非網格，理由：「工程要精度與可編輯性」，輸出 STEP）。有 Python/TS/Go/Rust client。
  • Spectral Labs SGS-1 ⚠ medium — 影像/網格 → STEP B-Rep，可在 SolidWorks/Fusion 開啟；但「parametric/first」宣稱有爭議，其效能 benchmark 本研究已駁回剔除。
  • BlenderMCP ✓ 可用 — MCP → Blender，執行 bpy Python(exec)、server 內零 canvas 操作。第三方評測：「可用工具非 demo」，但空間定位不精確。「MCP 串接 3D 軟體」的直接類比。
  • Adam CAD 商用 — YC、融資 $4.1M，從 text-to-3D 轉 AI copilot。
  • 學術（已發表/開源） 研究 — DeepCAD(首個把 CAD 表示成「操作序列」的生成模型)、Text2CAD(NeurIPS 2024，文字→序列化參數 CAD)、CAD-Coder(影像→可編輯 CadQuery 程式碼，100% 語法率、勝 GPT-4.5/Qwen-VL；屬主場 benchmark)。

③方法論比較與建議

為何「AI 點 GUI/canvas」難（本專案親身踩到）：精度／座標／即時操作不可靠 — WebGL 畫布 1px 邊線、合成點擊掉字、視窗座標漂移，全是死路。

為何「AI → 程式碼/API」可行：給出精確（曲線是真曲線）、可重現、可版本化、可被 LLM 自我檢視迭代的幾何定義。Onshape(MCP/OnPy)、Blender(bpy)、Zoo(KCL)、學術(序列化操作) 四個獨立領域證據完全收斂。

👉 對本專案的具體建議：放棄 WebGL 點選，改用 Onshape REST API（直接呼叫，或經現成 jarvis MCP server / OnPy），沿用「程式化下指令 ＋ 多視角算圖視覺驗證」迴圈。這會一次解決所有卡關。

⚠注意事項與限制

  • API 配額：Onshape 2025 起年配額低（Free/EDU 僅 2,500 次、Pro 5,000、Enterprise 每用戶 10,000；超額回 402），MCP server 的節流策略是否夠用尚無證據。
  • 品質落差：這些都證明「能做、用什麼方法」，但複雜幾何的可靠度都會掉；輸出品質證據偏軼事性（jarvis 自述 1 小時建支架、BlenderMCP 空間定位不精確）。
  • 效能數字須謹慎：SGS-1「勝 GPT-5」benchmark 已被駁回；CAD-Coder「100% 語法/勝 GPT-4.5」雖經同儕審查但屬作者自有測試集，跨分布泛化未證實。
  • 細節更正：jarvis/hedless 實際是 Onshape API 金鑰(access/secret)認證，非部分 README 說的 HMAC。
